# AI 面试分身引用区改造设计

## 1. 目标

本次改造的目标是收敛当前单页聊天界面的信息展示方式，移除右侧独立状态面板，将“引用来源”直接绑定到对应的 assistant 回答下方展示。

这样做的核心原因有两个：

1. 当前右侧 `TerminalStatusPanel` 展示的信息里，真正对面试官有价值的是回答所依据的引用来源，而不是独立存在的会话侧栏。
2. 引用信息天然属于某一条回答，把它挂在消息本身上，能够避免多轮追问时出现“当前来源属于哪条回答”不清晰的问题。

本次改造不追求重新设计整套聊天页，而是专注于完成以下收敛：

- 去掉右侧 `TerminalStatusPanel`
- 删除 store 中只为该面板服务的状态与赋值逻辑
- 将消息中的 `sources` 从轻量标签升级为完整引用对象
- 在 assistant 消息下方渲染引用区，展示来源名称、可点击链接、引用摘要/命中文本片段

---

## 2. 范围

### 2.1 本次包含

- 删除右侧 `TerminalStatusPanel` 组件及页面中的引用
- 调整聊天页布局为单列主聊天区
- 精简 interview store，删除仅为侧栏服务的状态
- 修改前端消息类型定义，升级 `sources` 结构
- 修改前端消息渲染，在 assistant 消息下方展示引用区
- 修改前端 API 类型，使其接收新的引用对象结构
- 同步调整后端返回契约，使每条 assistant 消息自带完整引用信息
- 补充前端测试，验证引用区展示与无引用时的隐藏逻辑

### 2.2 本次不包含

- 重新设计整个聊天页视觉风格
- 重新定义语音模式交互
- 新增独立的引用详情弹窗或跳转页
- 对检索算法本身做质量优化
- 引入兼容旧结构与新结构并存的过渡层

---

## 3. 现状问题

当前实现中，右侧状态面板承担了以下展示责任：

- 当前状态
- 当前话题
- 当前命中的来源标签
- 若干演示感装饰内容

对应代码中：

- `frontend/src/pages/interview/components/TerminalStatusPanel.vue` 负责右侧整块渲染
- `frontend/src/store/interview/index.ts` 中维护了 `topic`、`activeSources`、`displayStatus`
- `frontend/src/store/interview/types.ts` 中 `SourceTag` 仅包含 `label` 和 `kind`

这种结构的问题在于：

1. **引用语义不清晰**：来源标签挂在全局状态上，而不是某一条回答上，多轮对话后很难判断来源属于哪条消息。
2. **状态重复建模**：消息里已经有 `sources`，store 里又维护一份 `activeSources`，容易造成双份状态。
3. **信息价值不均衡**：右侧面板的大部分信息是展示型装饰，真正有解释价值的是回答引用。
4. **布局分散注意力**：对于面试场景，核心应该是“问题—回答—依据”，而不是让用户额外理解一个侧栏面板。

---

## 4. 设计原则

### 4.1 引用归属到消息

引用信息必须属于某一条 assistant 消息，而不是属于整个会话页面。

### 4.2 删除双份状态

如果某个信息已经能从消息流直接得出，就不要再在 store 中维护一份专门给 UI 面板使用的镜像状态。

### 4.3 优先最小改动路径

本次不做额外抽象层，不引入新状态容器，不保留新旧两套结构并存。直接切换到新的消息引用模型。

### 4.4 保持对话体验连续

删除右侧面板后，发送消息、语音切换、重置会话等主流程行为不能退化。

---

## 5. 页面与组件设计

### 5.1 页面结构调整

聊天页从“主聊天区 + 右侧状态面板”的双栏布局，调整为单列主聊天区。

页面中不再渲染 `TerminalStatusPanel`，也不再保留对应占位。

调整后的页面结构只保留：

- 顶部标题 / 模式切换区域
- 消息列表区域
- 输入与语音控制区域

### 5.2 删除的组件职责

`TerminalStatusPanel` 整体移除。以下内容不再以独立面板形式存在：

- `TELEMETRY_STREAM`
- `ACTIVE_PARAMETERS`
- `ACTIVE_SOURCES`
- preview / mini-map 类装饰区域
- 当前话题的侧栏展示

### 5.3 消息组件新增引用区

assistant 消息卡片底部新增“引用来源”区块。

展示规则：

- 仅 assistant 消息可显示引用区
- 仅当 `message.sources.length > 0` 时显示
- interviewer 消息永不显示引用区

每条引用展示三项内容：

1. **来源名称**
2. **可点击链接**
3. **引用摘要 / 命中文本片段**

推荐展示顺序：

- 第一行：来源名称
- 第二行：链接
- 第三行：摘要片段

推荐视觉形式：

- 在消息底部使用独立引用列表
- 每条引用用小卡片或边框列表项包裹
- 链接使用可识别的强调样式
- 摘要使用较弱层级文字，避免喧宾夺主

## 6. 前端数据结构设计

### 6.1 现有结构问题

当前 `SourceTag` 定义只有：

- `label`
- `kind`

这只适合做标签展示，不足以支持真正的引用信息展示。

### 6.2 新的引用对象结构

将 `InterviewMessage.sources` 的元素从轻量标签升级为完整引用对象。

推荐结构：

```ts
interface MessageCitation {
  title: string
  url: string
  snippet: string
  kind: string
}
```

`InterviewMessage` 保持“消息自带引用”的模型：

```ts
interface InterviewMessage {
  id: string
  role: 'interviewer' | 'assistant'
  content: string
  sources: MessageCitation[]
  createdAt: string
}
```

### 6.3 字段语义

- `title`：来源名称，例如文件名、知识条目标题、页面标题
- `url`：可点击链接，优先返回可直接访问的目标
- `snippet`：与本次回答最相关的命中文本片段或摘要
- `kind`：来源类型，用于后续保留样式或分组能力

### 6.4 不采用的方案

本次不采用独立 `messageCitations` 映射表，也不保留会话级 `activeSources` 作为引用数据源。

原因是当前需求只要求“引用属于某条回答”，没有必要为此引入额外一层映射结构。

---

## 7. Store 设计

### 7.1 保留的状态

`useInterviewStore` 继续保留真正驱动聊天主流程的状态，例如：

- `messages`
- `status`
- `voiceStatus`
- `mode`
- `draft`
- `speechEnabled`
- `currentSessionId`
- `send`
- `reset`

### 7.2 删除的状态

以下状态删除：

- `topic`
- `activeSources`

如果 `displayStatus` 只用于右侧面板展示，也应一并评估是否还保留；若页面其他位置仍依赖它展示语音 / 发送状态，则可保留其作为真实流程状态的派生值，而不是面板模型的一部分。

### 7.3 发送流程调整

当前发送流程在收到回复后，会额外执行：

- 更新 `topic`
- 更新 `activeSources`

改造后，收到回复时只需要将 reply 推入消息列表：

```ts
messages.value.push(payload.reply)
```

不再额外维护面板型来源状态。

### 7.4 重置流程调整

`reset()` 不再需要重置 `topic` 和 `activeSources`，只重置真实会话相关状态与欢迎消息。

---

## 8. API 与后端契约设计

### 8.1 前端 API 契约变化

前端 API 层不再把 `reply.sources` 视为标签数组，而要视为完整引用对象数组。

即前端消费目标从：

```ts
{ label, kind }[]
```

切换为：

```ts
{ title, url, snippet, kind }[]
```

### 8.2 后端返回要求

后端生成 assistant 消息时，返回的 `sources` 需要直接满足前端展示需要。

每条 assistant 消息应尽量返回：

- `title`
- `url`
- `snippet`
- `kind`

其中：

- `title` 必填
- `url` 目标上应尽量始终返回可用值
- `snippet` 应是本次回答实际引用的相关片段，而不是空字符串占位

### 8.3 不做双结构兼容

本次不建议前端同时兼容：

- 旧结构 `label/kind`
- 新结构 `title/url/snippet/kind`

原因是：

1. 当前需求本身就是一次明确收敛
2. 兼容层会引入过渡代码和字段分支
3. 对现阶段项目收益小，反而增加维护复杂度

因此推荐前后端一次性切换到新契约。

---

## 9. 渲染与交互细节

### 9.1 渲染条件

引用区的渲染条件如下：

- `message.role === 'assistant'`
- `message.sources.length > 0`

否则不渲染引用区。

### 9.2 链接行为

链接应以可点击方式展示。

推荐行为：

- 使用标准 `<a>` 链接
- 新标签页打开
- 附带安全属性

### 9.3 摘要展示

`snippet` 用于帮助面试官理解“这条回答引用了什么”，因此应允许展示多行文本，但默认只占消息底部一个较轻量区域。

如摘要较长，可通过样式进行截断或限制显示行数，但不在本次设计中引入展开 / 收起交互。

### 9.4 空链接处理

虽然这次目标是固定展示三项，但从工程角度仍应要求后端尽量保证返回完整字段，避免前端出现大量条件分支。

因此设计上以“后端总是尽量返回完整 `url`”为默认约束，而不是前端去容忍大量空值。

## 10. 测试设计

### 10.1 前端单测

至少覆盖以下场景：

1. **assistant 消息有引用时显示引用区**
   - 渲染来源名称
   - 渲染链接
   - 渲染摘要片段

2. **assistant 消息无引用时不显示引用区**

3. **interviewer 消息即使 sources 非空也不显示引用区**
   - 该场景主要用于保护渲染边界

### 10.2 Store 测试

至少验证：

1. 发送成功后仅追加消息，不再写入 `topic` / `activeSources`
2. `reset()` 后消息恢复欢迎态，且不存在被删除字段的重置逻辑依赖

### 10.3 页面集成验证

至少验证：

1. 删除右侧面板后页面仍能正常发送消息
2. assistant 回答出现后，引用区跟随该条消息展示
3. 会话多轮追问时，每条回答只展示自己的引用
4. 语音模式与文字模式切换不影响消息引用显示

## 11. 风险与控制

### 11.1 风险：后端未及时切换新结构

如果前端先改而后端仍返回旧 `label/kind` 结构，引用区将无法完整展示。

**控制方式：** 前后端在同一轮改造中同步切换，不做半兼容状态。

### 11.2 风险：删除侧栏后页面过于空旷

去掉右侧面板后，页面横向布局会发生变化。

**控制方式：** 通过主聊天区宽度、消息卡片层次、引用卡片样式补足信息密度，而不是再引入新的装饰侧栏。

### 11.3 风险：引用摘要过长干扰阅读

如果 `snippet` 很长，可能压过主回答内容。

**控制方式：** 样式上控制摘要层级与显示高度，保持它是“依据说明”，而不是主内容。

---

## 12. 最终决策

本次方案的最终决策如下：

1. 删除右侧 `TerminalStatusPanel` 组件及其页面引用
2. 删除 store 中仅服务于该面板的 `topic`、`activeSources`
3. 保留消息自带 `sources` 的模型，但将其升级为完整引用对象数组
4. assistant 消息下方新增引用区，展示三项内容：
   - 来源名称
   - 可点击链接
   - 引用摘要 / 命中文本片段
5. 前后端一次性切换到新的引用对象契约，不做双结构兼容

这是一条最小但语义完整的收敛路径：既去掉了当前分散注意力的右侧侧栏，又把真正重要的“回答依据”放回到回答本身下面，符合面试场景下“问题—回答—依据”的阅读顺序。
