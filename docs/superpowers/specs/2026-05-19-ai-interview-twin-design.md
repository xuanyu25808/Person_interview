# AI 面试分身单页设计说明

## 1. 项目目标

本项目不再以“个人中心多页面站点”为第一目标，而是收敛为一个更适合面试展示的单页产品：**AI 面试分身**。

目标是让面试官进入页面后，能够直接通过文字或语音与“电子版的我”进行面试式对话，快速了解我的项目经历、技术能力、设计思考与工程判断。

该产品强调：

- 单一强记忆点，而不是多模块分散展示
- 回答基于真实资料，而不是通用大模型自由发挥
- 支持面试场景中的连续追问，而不是一次性问答
- 在演示体验上兼顾视觉质感、响应流畅度与语音交互能力

---

## 2. 产品定位

这是一个**专用 AI 面试分身系统**，不是通用聊天助手，也不是多 Agent 平台。

系统的职责是：

- 以“我本人”的身份回答问题
- 优先依据我的真实资料进行回答
- 支持围绕我的项目和经历进行技术追问
- 在超出真实资料边界时拒绝编造

因此，该系统的产品定位应明确为：

> 一个支持文字与语音双模切换的 AI 面试聊天页，用于展示候选人的真实经历、项目能力与工程思考。

---

## 3. 第一版范围调整

### 3.1 不再优先建设的内容

相较于原始个人中心 Monorepo 方案，第一版不再优先建设以下能力：

- 多页面个人中心导航结构
- Projects / Notes / Status 多 Tab 站点外壳
- MCP 集群
- 多 Agent 编排系统
- 通用工具平台
- 长期用户记忆平台
- 复杂后台管理系统
- 多模型路由平台

### 3.2 第一版保留的核心能力

第一版只保留以下五类核心能力：

1. 双模对话：支持文字与语音切换
2. Persona 约束：固定为“电子版的我”
3. Interview RAG：基于真实资料进行检索增强回答
4. Session Memory：保持当前面试上下文连续性
5. Answer Guardrail：资料不足时不编造

---

## 4. 页面结构设计

整个产品为单页应用，建议采用深色、极简、专业的视觉风格，以凸显“AI 面试终端”的气质。

### 4.1 顶部区域

顶部区域包含：

- 姓名或标题
- 一句定位文案，例如：Talk to my AI interview twin
- 模式切换控件：文字模式 / 语音模式
- 简短使用说明

### 4.2 主聊天区域

主聊天区域用于展示完整问答消息流：

- 区分面试官消息与 AI 回答
- AI 回答支持 Markdown 呈现
- 可附带来源标签或关联项目标签
- 支持流式输出反馈

### 4.3 状态信息区域

状态区域用于提高可解释性与演示感，建议显示：

- 当前状态：空闲、监听中、识别中、检索中、思考中、回答中
- 当前面试主题
- 当前命中的项目或资料标签

### 4.4 底部输入区域

底部区域包含：

- 文字输入框
- 发送按钮
- 麦克风按钮
- 语音播报开关
- 清空会话或开始新一轮面试按钮

---

## 5. 交互流设计

### 5.1 文字模式

1. 面试官输入问题
2. 后端执行意图理解、上下文读取、资料检索与答案生成
3. 前端流式展示回答
4. 如有需要，展示相关资料标签

### 5.2 语音模式

1. 面试官点击麦克风开始说话
2. 火山引擎 ASR 实时转写语音内容
3. 转写结果进入统一问答主链路
4. LLM 生成回答
5. 前端显示回答文字
6. 火山引擎 TTS 播报回答

### 5.3 模式切换原则

文字模式与语音模式必须共用同一套：

- persona
- retrieval
- session memory
- answer policy
- 会话记录

也就是说，切换的只是交互方式，而不是两套不同的 AI 系统。

---

## 6. 后端能力设计

### 6.1 Persona 层

Persona 层用于定义“电子版的我”的身份边界，包括：

- 我的背景
- 我的项目经验范围
- 我的技术偏好
- 我的表达风格
- 哪些内容可以讲
- 哪些内容不能编

它的目标是保证系统回答始终像“我本人”，而不是普通通用助手。

### 6.2 Retrieval 层

Retrieval 层用于支持基于资料的回答。知识源建议只包含与候选人强相关的内容：

- 简历
- 项目介绍与项目拆解文档
- 技术笔记
- 博客或文章
- 常见面试问答素材
- 技术选型思考记录

检索原则为：

- 优先检索与当前问题最相关的真实资料
- 如果是项目追问，优先返回项目材料
- 如果是技术追问，优先返回与项目语境相关的资料
- 若资料不足，再允许模型做有限度的原理解释

### 6.3 Session Memory 层

Session Memory 第一版采用分层设计，不做通用长期记忆平台，但要体现对长上下文管理与知识沉淀的理解。

建议分为三层：

1. Recent Turns Buffer
   - 保留最近若干轮原始对话
   - 用于保持当前追问的即时连续性

2. Session Summary Memory
   - 每隔若干轮对当前会话生成摘要
   - 提炼当前在聊的项目、技术主题、已确认结论、待继续深挖的问题
   - 后续回答优先参考摘要，而不是无限堆叠完整历史消息

3. Selective Memory Write-back
   - 将高价值、稳定、可复用的回答片段结构化沉淀
   - 例如某项目亮点解释、某技术选型理由、某类常见追问的高质量回答
   - 不是自动全量写回，而是基于规则筛选后选择性回写到检索层

这样可以体现 memory 与 retrieval 的协同关系：

- Recent Turns 负责短期连续性
- Summary 负责压缩上下文成本
- Write-back 负责把高价值会话内容转化为可复用知识

该层的目标是保证多轮追问连续，并在长会话中维持人物一致性与回答稳定性，而不是构建复杂的通用记忆平台。

### 6.4 Answer Policy 层

回答规则建议分为三层：

1. 真实经历优先：资料中有明确依据时，按真实经历回答
2. 技术追问可解释：允许围绕真实经历补充技术原理说明
3. 超边界不编造：没有依据时明确说明边界，不伪造经验

这一层是项目可信度的关键。

### 6.5 Memory 与 Retrieval 协同策略

系统应明确区分 memory 与 retrieval，但允许二者协同工作。

- retrieval 负责候选人的静态资料与已沉淀知识
- memory 负责当前会话中的短期上下文、摘要状态与候选沉淀内容

推荐策略：

1. 回答前优先读取 recent turns 与 session summary
2. 检索候选人的静态资料
3. 如有必要，再检索历史沉淀的 memory knowledge
4. 生成回答后判断是否需要：
   - 更新 session summary
   - 生成结构化 memory candidate
   - 将高价值内容选择性写回检索层

这样既能保证长会话连续性，也能体现对上下文压缩和知识沉淀机制的工程理解。

### 6.6 统一回答链路

统一主链路为：

输入问题 → 识别意图 → 读取会话上下文 → 检索资料 → 组装 persona 与上下文 → 生成回答 → 输出文本 / 语音

该系统不采用复杂多 Agent 编排，而采用受约束的 interview-answer pipeline，以提升稳定性、可控性与可解释性。

---

## 7. 技术架构建议

推荐采用“主后端 + 语音适配层”的架构。

### 7.1 推荐结构

- `frontend`：前端单页聊天界面
- `backend`：主问答后端
- `services/voice`：基于现有火山引擎实时语音项目改造的语音适配层
- `knowledge/` 或后端内部数据目录：候选人资料库

### 7.2 职责划分

#### 前端

负责：

- 页面展示
- 聊天消息流
- 模式切换
- 状态反馈
- 语音控制入口

#### 主后端

负责：

- 会话管理
- persona 组装
- retrieval 调用
- session memory 管理
- answer generation
- 流式结果输出

#### 语音适配层

负责：

- ASR 语音识别
- TTS 语音播报
- 与现有火山语音项目对接

### 7.3 不推荐的第一版架构

第一版不建议直接拆分为：

- 检索服务
- 记忆服务
- MCP 集群
- 多 Agent 协作层
- 通用工具注册系统

原因是当前问题域足够聚焦，过早服务化不会提升展示效果，反而增加联调与稳定性风险。

---

## 8. 数据来源边界

第一版知识源建议仅包含：

- 简历
- 项目材料
- 技术笔记
- 提前整理的面试问答素材

不建议第一版直接接入过多杂乱来源，如：

- 所有仓库全文
- 零散聊天记录
- 无结构网页资料
- 大量无筛选外部文档

第一版更重要的是保证回答稳定、像本人、可验证。

---

## 9. 第一版最小可交付标准

### 9.1 页面侧

- 单页视觉完成度高
- 文字模式稳定可用
- 语音模式可演示
- 模式切换清晰
- 状态反馈完整

### 9.2 AI 侧

- 能正确回答自我介绍类问题
- 能回答项目经历与技术选型问题
- 能承接 2 至 4 轮连续追问
- 资料不足时不乱编
- 能围绕真实经历解释相关技术原理

### 9.3 工程侧

- 前后端边界清晰
- 语音链路与问答链路职责明确
- 知识源组织合理
- 演示时稳定，不容易现场翻车

---

## 10. 面试展示叙事建议

该项目在面试中最值得讲的技术判断有四个：

1. 为什么只做一个页面
   - 因为希望把展示重点收敛到一个强记忆点：AI interview twin

2. 为什么必须做 RAG
   - 因为目标是让系统基于真实资料回答，而不是做通用问答

3. 为什么只做轻量 session memory
   - 因为当前场景需要的是多轮追问连续性，而不是长期记忆平台

4. 为什么不做 MCP 集群或多 Agent
   - 因为这是专用场景，复杂编排不会直接提升展示效果，反而增加系统不稳定性

---

## 11. 对原计划的调整结论

原始“个人中心系统 Monorepo”方案适合作为长期扩展方向，但对于当前面试展示目标，第一阶段应收敛为单页 AI 面试分身。

因此，建议将当前阶段目标由：

- 构建多模块个人中心主壳

调整为：

- 构建一个支持文字与语音切换、具备 persona + RAG + session memory 的 AI 面试聊天单页

后续如果该单页产品验证效果好，再考虑把 Projects、Notes 等内容作为知识源扩展，而不是优先做多页面壳层。
